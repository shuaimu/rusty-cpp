# Stdlib-port regen un-rot — status & method

## Problem

The vendored `transpiled/*_port/*.cppm` are generated by
`rust-src subtree → docs/<port>/prep.sh → transpiler → docs/<port>/post_transpile_patch.py`.
For a long time none of the three inputs were pinned, so a regen stopped
reproducing the vendored ports:

- **source** was a moving `stable` (no pin);
- the **transpiler** changed enormously (UFCS migration, inference engine,
  a newer embedded prelude) so its emission drifted from what the patchers'
  text anchors expect;
- the **patchers** went unmaintained — across all 23 ports there was **1**
  `# RETIRED:` marker total, despite the convention to retire redundant
  patches;
- **hand-fixes leaked** straight into the vendored `.cppm` without being
  back-ported to the patcher (e.g. vec_port `clone_into`, commit 56fc9428:
  a regen reproduced the empty-stub no-op that had caused a double-free).

Verified on vec_port (2026-07-11): a full regen produced **18** modules vs
the vendored **6**, ~1372 differing lines in `vec.cppm`, `clone_into` reverted
to `{}`, and 13 patches "skipped" (ambiguous: already-applied *or*
anchor-drifted-no-op). See memory `port-regen-pipeline-bitrot`.

## Pin

`rust-toolchain.toml` now pins **1.95.0** (the version the ports were first
vendored against, 2026-05-30). This fixes the stdlib SOURCE. Bumping the
channel = re-baselining every port; do it deliberately.

## Method: un-rot = RE-BASELINE (not reproduce-the-old)

Most of a port's regen-vs-vendored diff is the transpiler emitting the *same*
methods with newer/equivalent syntax plus a newer embedded prelude — i.e. the
transpiler improved. Forcing the old emission back would mean huge adversarial
patches for no benefit. So per port:

1. `docs/port_regen/regen_diff.sh <port>` — regenerate + patch + diff.
2. Inspect `only_vendored.txt` for genuine **lost hand-fixes** (port logic /
   safety fixes the recipe fails to reproduce) and **back-port each into the
   patcher** (like `clone_into`). Ignore stale-prelude + resyntax noise.
3. Get the fresh regen to **compile** (fix the patcher's module-merge so the
   `.cppm` set matches, retire dead patches).
4. **Matrix-validate**: the regenerated port must pass the parity matrix
   (its own crate + every crate that links it — vec/btree/indexmap/serde…).
5. **Replace** `transpiled/<port>/*.cppm` with the validated regen. Now
   `regen == vendored` holds by construction.
6. Eventually: a CI guard running `regen_diff.sh` that fails if
   `only_vendored.txt` has real (non-prelude) content — prevents re-rot.

## vec_port re-baseline — in progress (resume here)

Recipe now takes the fresh regen from wildly-broken to **5 compile errors**,
all in `vec_port.vec.into_iter.cppm`. Patcher gained (this session):
`clone_into` real body; `std::hint::`/`std::__builtin_assume` handling in
`patch_hint_assert_unchecked`; and `patch_current_transpiler_qualification_drift`
(strip `rusty::`-qualified keywords `reinterpret_cast`/`sizeof`, collapse
`rusty::rusty::`, `rusty::ptr::cast(p)`→`p.cast()`, drop the circular
`import rusty;` from port modules).

**into_iter.cppm: 5 errors RESOLVED** (both drifted stub anchors, now
regex-tolerant): `into_vecdeque` (VecDeque arity — deliberate stub, see
below) and the `super::Vec::new_in` Default (pre-existing single-site
`super::` non-resolution — stubbed). NOTE on into_vecdeque: it backs the
PUBLIC `VecDeque::from(vec)` conversion; the stub makes that abort. It is
stubbed for a real reason — VecDeque depends on Vec (not vice versa), so
vec_port can't construct a VecDeque without inverting the dep / cycling.
Proper fix (if Vec→VecDeque is ever needed) lives in vec_deque_port.

**vec.cppm: ~20 MORE drift errors (next, the real remaining grind):**
- 9× `no template named 'Vec' in namespace 'rusty'` (rusty::Vec over-qual → `::Vec`/`Vec`)
- 4× `std::rusty::…` (std:: mis-qualification)
- 2× `rusty::intrinsics::aggregate_raw_ptr` missing
- 1× each: `ret` undeclared, `IntoIter` return-type deduction, `std::hint`
Same over-qualification family as raw_vec — a few more targeted patcher rules.

**TRIAGE of the over-qualification errors (2026-07-11) — patcher vs transpiler:**
- **OPTION C (general transpiler bug, do FIRST):** `std::rusty::` / `rusty::rusty::`
  in EXPRESSION position (`std::rusty::slice_ext::range(...)`, `std::rusty::iter_ext::zip(...)`).
  The transpiler ALREADY has `collapse_duplicate_rusty_root_namespaces_in_type`
  (mod.rs:39385) that fixes these — but it's called from ONE type-emission site
  (mod.rs:39356), so expression-position occurrences leak. FIX: make that collapse
  a universal final pass. Low risk (matrix crates don't emit these in exprs or
  they'd fail). Deletes this class from ALL 13 ports at once. Gate against full matrix.
- **PATCHER (port-local):** `rusty::Vec` self-qual (Vec port qualifying its own Vec —
  normal crates correctly want rusty::Vec); `aggregate_raw_ptr` (existing rule, anchor
  drifted to new intrinsic name — update anchor).
- **REVISIT at next port:** `rusty::`+keyword (reinterpret_cast/sizeof) — normal cast
  path emits bare `reinterpret_cast` correctly (emit_expr.rs:8188/9103), so this is a
  narrower path; patched port-local, promote to Option C only if it recurs.

Then: get into_iter clean → **matrix-validate** (vec + btree + indexmap +
